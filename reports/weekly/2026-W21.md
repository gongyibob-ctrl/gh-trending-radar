# 本周雷达 — 2026-W21

## 一句话总览
Agent 时代缺的是控制面。

## 三个值得深挖的方向

## 1. **给编码 Agent 的“安全工作台”：沙箱、并行、合并、审计**

**数据支撑**

- [mattpocock/sandcastle]：TypeScript 库，编排 sandboxed coding agents，518 forks。
- [superset-sh/superset]：Launch HN，定位 “IDE for the agents era”，121 条评论。
- [48225040 Runtime]：YC 项目，给团队提供 sandboxed coding agents，强调非工程师也能安全操作代码库。
- [1144878 TestSprite 3.0]：用一组并行 agents 在几分钟内测试 app。

**为什么这是机会**

现在不是“再做一个 Copilot”的窗口，而是“让团队敢把 agent 接进真实代码库”的窗口。

缝隙在三处：

1. **现有 coding agent 太个人化**：Claude Code / Antigravity / Cursor 更像个人生产力工具，团队层面的权限、审计、回滚、成本控制不足。
2. **并行 agent 带来新复杂度**：多分支、多任务、多 agent 输出后，谁负责冲突解决、质量门禁、测试覆盖、PR 解释？这还没有成为标准工作流。
3. **非工程师想用，但风险更高**：Runtime 明确打 “everyone on a team”，说明需求在从工程师扩展到 PM、运营、客服、销售工程师，但他们不能直接碰生产代码。

这类机会更像“Agent DevOps / Agent SDLC”，不是 IDE 本身。

**与他的 SaaS 背景的契合点**

契合点：

- 如果他做过 B2B SaaS，天然理解团队协作、权限、审计、工作流、审批，这比纯开发者工具更接近 SaaS 能力。
- 可以从“围绕现有 Claude Code / GitHub / Linear / Slack 的控制层”切入，而不是重做底层 agent。

需要补的新能力：

- 深度理解 Git 工作流、CI/CD、安全沙箱、repo 权限模型。
- 要能服务开发团队，产品体验需要非常工程化，不能只做漂亮 dashboard。

**立即可做的小动作**

- 访谈 5 个已经在团队里用 Claude Code / Cursor / Devin 类工具的工程负责人，问清楚：他们现在最怕 agent 改坏什么？如何 review？如何回滚？
- 做一个极小原型：连接 GitHub + Claude Code 输出 + CI，把 agent 生成的变更自动放进隔离分支，并生成“风险摘要 + 测试建议 + reviewer checklist”。

## 2. **Agent 权限与身份层：邮箱、密钥、OAuth、可撤销授权**

**数据支撑**

- [48225596 Agent.email]：给 AI agents 自己的邮箱和注册流程，98 条 HN 评论。
- [1152914 DCP]：给 AI agents 加密权限和 keys。
- [48239786 Domain-Camouflaged Injection Attacks]：多 agent 系统里的 prompt injection 新攻击面。
- [mukul975/Anthropic-Cybersecurity-Skills]：754 个结构化网络安全 skills，映射 MITRE / NIST，1017 forks。

**为什么这是机会**

Agent 正在从“聊天窗口”变成“能登录、收邮件、调 API、访问 SaaS、执行任务”的数字员工。问题是：今天的 SaaS 身份体系默认用户是人，不是 agent。

缝隙很明确：

1. **账号体系不适配 agent**：agent 需要邮箱、OTP、API token、OAuth consent，但现有注册/登录/权限流程全是给人设计的。
2. **密钥管理太粗糙**：现在很多团队把 token 塞进环境变量或 prompt 上下文，缺少最小权限、过期、作用域、审计。
3. **安全风险从“模型回答错”升级为“agent 做错事”**：prompt injection 一旦能触发真实权限，就不只是幻觉问题，而是资产和数据风险。

这是一个适合做基础 SaaS 的位置：Agent IAM / Agent Secret Manager / Agent Access Gateway。

**与他的 SaaS 背景的契合点**

契合点：

- B2B SaaS 创始人通常懂多租户、RBAC、审计日志、企业采购、安全合规，这些正是 agent 权限层的卖点。
- 可以从“给现有 SaaS/内部工具加 agent-safe access”切入，而不是教育市场买全新平台。

需要补的新能力：

- 安全产品门槛较高，需要懂 OAuth/OIDC、密钥托管、权限边界、合规语言。
- 销售上会面对 CTO / Security / Platform team，销售周期可能比普通 SaaS 长。

**立即可做的小动作**

- 访谈 3 类人：DevOps、内部工具负责人、安全负责人。核心问题：你们是否允许 agent 使用生产 SaaS token？如果不允许，卡在哪里？
- 做一个 “Agent OAuth Proxy” demo：给 agent 发放短期、可撤销、scope-limited token；每次调用 Slack / Linear / GitHub 都有审计记录和人工确认选项。

## 3. **Agent 可读的业务数据层：让产品、分析、客户反馈能被 agent 调用**

**数据支撑**

- [1152064 Mixpanel Headless]：给 agents 和 developers 提供 product analytics 的 programmatic access。
- [1153040 DecisionBox for Databricks]：连接 Databricks 验证分析发现。
- [1152985 Nugget AI]：把客户访谈转成产品 roadmap。
- [1146304 CatchAll by NewsCatcher]：按条件从 Web 构建数据集。

**为什么这是机会**

本周一个低调但重要的信号：SaaS 正在从“给人看的 UI”变成“给 agent 调用的能力”。Mixpanel Headless 是最直接的例子：产品分析不只是 dashboard，而是 agent 可以查询、解释、行动的 API。

缝隙在于：

1. **业务数据分散**：客户访谈在 Gong/Zoom/Notion，产品数据在 Mixpanel/Amplitude，工单在 Zendesk，销售记录在 CRM。Agent 要做 PM 或增长助手，必须跨源拿数据。
2. **现有 BI 不负责“行动闭环”**：BI 给图表，roadmap 工具给任务，客户反馈工具给摘要，但很少能完成“发现问题 → 验证数据 → 生成决策建议 → 推进执行”。
3. **企业不信纯生成式洞察**：DecisionBox 强调 validate findings，说明市场已经意识到“AI 总结”不够，必须能追溯证据、校验口径。

机会不是再做一个“AI 数据分析聊天框”，而是做 **agent-ready business context layer**：把产品数据、客户声音、运营指标变成可查询、可验证、可行动的上下文。

**与他的 SaaS 背景的契合点**

契合点：

- SaaS 创始人天然懂产品指标、客户反馈、roadmap、增长漏斗，这比纯技术团队更懂业务闭环。
- 如果过去有 PM / CRM / 客户成功 / 数据分析相关经验，可以很快找到高频场景。

需要补的新能力：

- 数据连接器和权限模型会比较重。
- 必须建立“可信输出”能力：引用来源、口径解释、置信度、反例检查，而不是只做 summary。

**立即可做的小动作**

- 访谈 5 个 B2B SaaS 的 Head of Product / Growth：他们每周如何把客户反馈、产品数据、销售反馈合成 roadmap？最耗时间的环节是什么？
- 做一个窄原型：接入 Mixpanel + Intercom/Zendesk + Notion，回答一个具体问题——“过去 30 天流失客户最常提到的 3 个产品问题，是否在行为数据中有证据？”

## 反直觉的 1 个观察

模型更便宜了，但“AI 是否划算”讨论更热了。

一边是 [48237663 DeepSeek] 把 V4 Pro API 价格 75% 折扣永久化，249 条评论；另一边 [48243863 Is AI Profitable Yet?] 讨论 AI 商业是否盈利，194 条评论；还有 [48244434 Microsoft reports AI is more expensive than paying human employees] 指向 enterprise agent 成本压力。

这说明机会不一定在“更便宜的推理”本身，而在 **agent 成本治理**：任务级成本归因、模型路由、缓存、失败重试控制、ROI 追踪。Agent 越自动化，越容易把 token 成本、工具调用成本、人工 review 成本隐藏起来。企业真正会买的是“可控、可证明划算”，不是“又一个更聪明的 agent”。
