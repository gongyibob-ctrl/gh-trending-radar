# 本周雷达 — 2026-W20

## 一句话总览
Agent 时代先缺“管控层”。

## 三个值得深挖的方向

### 1. **AI Coding Agent 的记忆、技能与质量门禁**

**数据支撑**

- [rohitg00/agentmemory]：面向 Claude Code、Codex 等 coding agents 的持久记忆层，GitHub forks 934。
- [multica-ai/andrej-karpathy-skills]：一个 `CLAUDE.md` 文件改进 Claude Code 行为，forks 13,666，说明“可复用 agent 操作规范”需求极强。
- [addyosmani/agent-skills]：把生产级工程流程包装成 agent skills，forks 4,709。
- [millionco/react-doctor]：直接定位“Your agent writes bad React. This catches it”，说明 AI 写代码后的质量校验正在变成刚需。

**为什么这是机会**

现在 coding agent 已经从“能不能写代码”进入“能不能稳定交付”的阶段。缝隙在于：

- Agent 记不住项目长期上下文，跨会话、跨工具协作差。
- 团队缺少可审计、可复用、可版本化的 agent 工作规范。
- AI 生成代码质量不稳定，现有 lint / test / review 工具不理解“agent 产生的风险模式”。
- 企业不会只买一个更聪明的 agent，而会买“让 agent 可控、可复盘、可上线”的治理层。

这是典型的“卖铲子”机会：不与 Claude Code、Cursor、Codex 正面竞争，而是做它们上面的团队级控制面。

**与他的 SaaS 背景的契合点**

契合点：

- SaaS 创始人通常熟悉权限、团队空间、审计日志、用量计费、集成、协作流，这些正是 agent 管控产品需要的能力。
- 可以从开发团队切入，销售路径比纯 infra 更清晰：工程负责人、CTO、技术团队预算。

需要补的新能力：

- 需要懂开发者工作流，尤其是 repo indexing、CI/CD、PR review、agent prompt / skill 生命周期。
- 产品体验要足够轻，不能做成另一个重型 DevOps 平台。

**立即可做的小动作**

1. 访谈 8–10 个已经在团队中使用 Claude Code / Cursor / Codex 的工程负责人，问三个问题：  
   - AI 写代码后最常出什么事故？  
   - 团队有没有共享 `CLAUDE.md` / rules / prompts？  
   - 他们是否愿意为“agent 产出质量报告 + 风险拦截”付费？
2. 做一个极小原型：GitHub App，扫描 PR，标记“疑似 agent 生成代码风险”：未覆盖测试、重复逻辑、React anti-pattern、破坏现有约定，并生成 reviewer checklist。

### 2. **企业 Agent 的连接器与动作执行层**

**数据支撑**

- [1147255] Basedash MCP Connectors：主张“Connect any app and take action anywhere”，把 MCP 作为业务系统动作层。
- [1146408] TrustClaw by Composio：自托管 AI agent，连接 1000+ apps，部署在 Vercel。
- [HKUDS/CLI-Anything]：目标是“Making ALL Software Agent-Native”，forks 3,471，把既有软件暴露成 agent 可调用接口。
- [1147371] Agent-Sin：通过 reusable skills 处理重复业务任务，说明非开发者场景也在靠“技能化”落地。

**为什么这是机会**

Agent 真正进入公司流程，卡点不是模型，而是：

- 业务系统太碎：CRM、工单、Notion、Slack、数据库、内部后台。
- 连接器能“读”不够，企业真正要的是“安全地写入 / 执行动作”。
- Zapier / Make 偏确定性自动化，适合规则明确的流程；agent 适合半结构化任务，但缺权限、审批、回滚、日志。
- MCP 正在成为事实接口标准，但企业缺“可管理的 MCP 层”。

缝隙在“安全动作层”：让 agent 能操作 SaaS，但每一步有权限边界、审批流、审计、dry-run、回滚建议。

**与他的 SaaS 背景的契合点**

契合点：

- 如果他有 B2B SaaS 经验，应该熟悉客户最痛的不是“想不想自动化”，而是“谁能动哪些数据、出了错谁负责”。
- SaaS 背景适合做垂直切入：先选一个高频业务流，而不是做泛连接器平台。

需要补的新能力：

- MCP / OAuth / 权限模型 / 企业安全要求。
- 需要强集成能力，早期最好别贪多，选 3–5 个核心系统打穿。

**立即可做的小动作**

1. 选一个垂直流程做访谈，比如：  
   - 客服：agent 读工单、查 CRM、起草退款、需要人工批准后执行。  
   - 销售运营：agent 清洗线索、补全公司信息、更新 HubSpot / Salesforce。  
2. 做一个“只读 + 待审批写入”的 demo：agent 给出行动计划，用户点击批准后执行，并自动生成审计日志。

### 3. **AI 订阅与模型调用成本治理**

**数据支撑**

- [48168056] HN 热帖 “Every AI Subscription Is a Ticking Time Bomb for Enterprise”，277 comments，说明企业 AI SaaS / API 成本焦虑已经显性化。
- [decolua/9router]：多模型路由、fallback、节省 token，forks 1,752。
- [Alishahryar1/free-claude-code]：把 Claude Code 路由到替代或本地模型，forks 3,756，反映开发者对成本和限制的强烈绕行需求。
- [colbymchenry/codegraph]：本地代码知识图谱，目标是减少 token 和 tool calls，forks 250。

**为什么这是机会**

企业正在同时购买 ChatGPT、Claude、Cursor、Copilot、各种 AI SaaS，再叠加 API 调用费。问题不是单个工具贵，而是：

- 采购分散，团队自己刷卡，财务不可见。
- 模型调用缺少成本归因：不知道哪个团队、哪个 workflow、哪个 agent 在烧钱。
- 现有 SaaS spend management 工具懂订阅，不懂 token、模型路由、上下文缓存、agent tool call。
- 开发者已经用开源 proxy / router 自救，但企业需要合规、可审计、可策略化的版本。

缝隙是“AI FinOps for agent workflows”：不仅统计账单，还能给出策略：何时用小模型、何时走本地、何时缓存、何时禁止高价模型。

**与他的 SaaS 背景的契合点**

契合点：

- 这是很 SaaS-native 的问题：dashboard、预算、团队管理、告警、权限、报表、集成账单。
- 可以卖给 CTO / CFO / RevOps，价值容易量化：节省 AI 成本、减少失控订阅。

需要补的新能力：

- 要接入主流模型供应商、AI IDE、agent runtime 的使用数据。
- 需要理解 token economics 和模型路由策略，不能只做普通 spend dashboard。

**立即可做的小动作**

1. 访谈 5 个 50–500 人公司的 CTO / Finance Ops，问：  
   - 现在有多少 AI 工具订阅？  
   - 有没有团队级预算？  
   - 是否知道 Claude Code / Cursor / API 每月具体烧在哪里？
2. 做一个轻量 MVP：接入 OpenAI / Anthropic / Cursor / GitHub Copilot 账单或导出文件，生成团队维度 AI 成本报告，并给出 3 条节省建议。

## 反直觉的 1 个观察

本周最强信号不是“更强的 agent”，而是围绕 agent 的“小文件、小规范、小控制层”。

比如 [multica-ai/andrej-karpathy-skills] 本质只是一个 `CLAUDE.md`，但 forks 达到 13,666；[mattpocock/skills] forks 7,729；[addyosmani/agent-skills] forks 4,709。说明市场正在从“模型崇拜”转向“操作系统化”：谁能把人的工程判断、流程约束、团队规范包装成 agent 可执行资产，谁就有机会成为下一层基础设施。

这对创业判断很重要：不要急着做一个“全能 AI 员工”，先做让现有 AI 员工少犯错、可复用、可管理的窄层产品。
